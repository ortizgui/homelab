# Context Mesh Framework

## Overview
Context Mesh is a 3-step documentation framework: **Intent → Build → Learn**.

### Step 1: Intent
- Understand WHAT and WHY before HOW
- Create feature files that describe business/user value
- Create decision files that record technical choices

### Step 2: Build
- Create Architecture Decision Records (ADRs) BEFORE implementation
- Follow documented patterns
- Respect decisions

### Step 3: Learn
- Update changelog after changes
- Record outcomes in decision files
- Create learning documents for significant insights

## File Type Separation (CRITICAL)

| File Type | Location | Purpose | What Goes In |
|-----------|----------|---------|--------------|
| Feature | `intent/feature-*.md` | WHAT the feature does, WHY it exists | Business value, user needs, acceptance criteria |
| Decision | `decisions/*.md` | Technical choices | Libraries, frameworks, approaches, rationale |
| Pattern | `knowledge/patterns/*.md` | How to implement | Code examples, reusable structures, conventions |
| Anti-Pattern | `knowledge/anti-patterns/*.md` | What to avoid | Mistakes, pitfalls, unwanted patterns |

**Content MUST NOT mix concerns:**
- Feature files: NO technical implementation details, NO code examples, NO library names
- Decision files: ONLY technical choices, rationale, alternatives
- Pattern files: ONLY code examples, how-to, implementation details

## Plan, Approve, Execute (MANDATORY)
Before any implementation:
1. **Plan**: Create or review ADR first
2. **Approve**: Confirm the plan with user
3. **Execute**: Implement following documented patterns

## When to Create vs Not Create Files

**Create files when:**
- New feature is identified
- Technical decision is made
- Reusable pattern is discovered
- Anti-pattern is identified

**Do NOT create files for:**
- Obvious conventions (use 2-space indent in Python)
- Trivial implementation details
- Temporary workarounds

## Bidirectional Links (MANDATORY)
- Feature MUST link to decisions: `[Decision: Name](../decisions/N-name.md)`
- Decision MUST link back to features: `[Feature: Name](../intent/feature-name.md)`
- Patterns MUST link to decisions: `[Decision: Name](../decisions/N-name.md)`
- Features CAN link to patterns: `[Pattern: Name](../knowledge/patterns/name.md)`

## Status Sections
Every file MUST include a Status section at the end:
- **Created**: Date of creation
- **Status**: Active, Deprecated, Superseded, Proposed
- **Phase**: Intent, Build, or Learn

## Definition of Done
- [ ] ADR exists before implementation
- [ ] Code follows documented patterns
- [ ] Decisions respected
- [ ] Context updated to reflect changes
- [ ] Changelog updated

## Context Structure
```
context/
├── .context-mesh-framework.md    # THIS FILE - framework rules
├── intent/                       # What & Why (business features)
│   ├── project-intent.md
│   └── feature-*.md
├── decisions/                    # Technical choices (HOW)
│   └── *.md
├── knowledge/                    # Patterns & anti-patterns
│   ├── patterns/
│   └── anti-patterns/
├── agents/                       # Agent-specific instructions
└── evolution/                    # Change log
    └── changelog.md
```
